### What's identical across `UnifiedCard` and `ListingCard`

Both files contain a **verbatim copy** of the premium/elite border logic (≈20 lines):

```ts
// Duplicated in UnifiedCard.tsx:77-90 AND ListingCard.tsx:81-90
const plan = typeof item.attributes?.plan === 'string' ? item.attributes.plan.toUpperCase() : null
const isElite = plan === 'ELITE'
const isPremium = isElite || plan === 'PREMIUM' || plan === 'FEATURED' || item.attributes?.isPremium === true
const frameClass = isElite
  ? 'border-amber-200/60 shadow-[0_0_0_1px_rgba(245,158,11,0.10),...'
  : isPremium
  ? 'border-slate-200/80 shadow-[0_0_0_1px_rgba(148,163,184,0.10),...'
  : '...'
const shineClass = isElite ? 'before:pointer-events-none ...' : ...
```

Both files have identical `CATEGORY_ICON` and `BADGE_INTENT` maps.

### What's different

- `UnifiedCard` — vertical (grid) layout, 16:10 image, thumbnail row below image
- `ListingCard` — horizontal (row) layout, fixed 280 px image column, mobile CSS `transform:scale()` trick

### `JobCard` duplication

`features/listings/components/JobCard.tsx` (takes `JobCardItem`) and `features/jobs/components/JobCard.tsx` (takes raw `JobItem`) are **two separate implementations** of the same card. The listings version is more polished (salary display, tags, time-ago) but is stateless (actions are passed as props). The jobs version fires API mutations internally. This is a **P2 unification candidate**.

### `TransportRequestCard` gap

`TransportRequestCard` uses its own inline `STATUS_CLASSES` and `STATUS_LABELS` maps instead of the shared `StatusBadge` component from `components/ui/badges/`. It should use `StatusBadge`.

### Recommendation

Extract the premium/elite frame logic to a shared utility:

```ts

  const plan = ...
  return { isElite, isPremium, frameClass, shineClass }
}
```

Merge `JobCard` duplicates: keep the polished stateless version in `features/listings/components/JobCard.tsx` and pass actions as props from the jobs page.

### Identical endpoint pattern

```ts
// buses.ts
export function useBusListings(params?, enabled = true) {
  const qs = new URLSearchParams(params).toString();
  return useQuery({ queryKey: ['buses', params], queryFn: () => apiRequest(`/buses${qs ? `?${qs}` : ''}`) });
}

// parts.ts — identical structure
export function useParts(params?, enabled = true) {
  const qs = new URLSearchParams(params).toString();
  return useQuery({ queryKey: ['parts', params], queryFn: () => apiRequest(`/parts${qs ? `?${qs}` : ''}`) });
}

// services.ts — identical structure
export function useCarServices(params?, enabled = true) {
  const qs = new URLSearchParams(params).toString();
  return useQuery({ queryKey: ['services', params], queryFn: () => apiRequest(`/services${qs ? `?${qs}` : ''}`) });
}
```

**5 list hooks** follow this exact pattern. A generic factory could replace all of them:

```ts
// Proposed: lib/api/listing-hooks.factory.ts
export function createListingHooks<T, S>(endpoint: string, queryKey: string) {
  return {
    useList: (params?: Record<string, string>, enabled = true) => useQuery<{ items: T[]; meta: PaginatedMeta }>(...),
    useSingle: (id: string) => useQuery<T>(...),
    useMy: () => useQuery<T[]>(...),
    useCreate: () => useMutation<T, unknown, Record<string, unknown>>(...),
    useUpdate: () => useMutation<T, unknown, { id: string; data: Record<string, unknown> }>(...),
    useDelete: () => useMutation<void, unknown, string>(...),
  }
}

export const busApi = createListingHooks<BusListingItem>('/buses', 'buses')
export const partsApi = createListingHooks<SparePartItem>('/parts', 'parts')
export const servicesApi = createListingHooks<CarServiceItem>('/services', 'services')
```
